# 2.1 Admin Panel — Core CRUD

**Phase:** 2 — Admin Panel

**Goal:** Dev-only local admin panel with full CRUD for all content collections. Runs at `/admin` routes in dev mode only.

## User Stories

- As a content owner, I want to add a new bear through a form instead of manually creating markdown files.
- As a content owner, I want to edit any bear's details without opening a text editor.
- As a content owner, I want to delete bears/products/events that are no longer needed.
- As a content owner, I want to browse all content with search and filtering.
- As a content owner, I want dropdown selectors for brands, species, and categories so I don't make typos.
- As a content owner, I want to upload a photo and have it automatically placed in the right folder with the right filename.
- As a developer, I want the admin panel to only exist in dev mode — not in production builds.

## Technical Approach

- Dev-only Astro routes under `/admin/*` (excluded from production build)
- **Preact** for interactive form components (islands architecture — only loaded on admin pages, zero production weight)
- SSR pages that read/write directly to `src/content/` on the filesystem
- Uses the same Zod schemas from `config.ts` for validation
- No authentication (local only — accessed via `npm run dev`)
- Minimal clean styling (not themed, just functional and pleasant)
- Changes saved to disk → you commit via git when ready

### Why Preact?

The admin panel needs interactive forms (dropdowns, file uploads, dynamic fields, preview toggles, geocoding search). Pure server-rendered pages would require full page reloads on every action. Preact provides:

- React-compatible syntax (3KB, tiny)
- Astro island architecture — only loads on admin pages
- Not included in production builds (admin routes excluded)
- Handles form state, validation feedback, and dynamic UI cleanly

## File Operations

When saving a content entry, the admin panel:

1. **Validates** all fields against the Zod schema
2. **Creates/updates** the directory (e.g., `src/content/bears/{slug}/`)
3. **Writes** the `index.md` file with frontmatter + body
4. **Copies/moves** uploaded images to the content directory, renamed to match the slug
5. **For photo processing:** resizes images to a sensible max dimension (e.g., 1200px wide) to avoid massive source files

When deleting:

1. **Removes** the entire content directory (folder + markdown + images)
2. **Warns** if other content references this entry (e.g., deleting a location used by bears)

## List Views

Each collection list page shows:

- Searchable, filterable table of entries
- Key fields visible at a glance (bear: name, species, brand, status; event: name, date, upcoming/past)
- Edit and delete buttons per row
- "Add new" button at top
- Count of total entries and filtered results
